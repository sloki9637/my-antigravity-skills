## 🎯 작성 목표

구글 애드센스 "가치 없는 콘텐츠" 정책 통과를 목표로 한다.
E-E-A-T(경험·전문성·권위·신뢰) 기준을 완전히 충족하며,
동일 주제를 검색한 독자가 이 글 하나로 완전히 해결되어야 한다.

글 유형: {TYPE_A / TYPE_B / TYPE_C 중 선택}
글 주제: {직접 입력}

## ⛔ 전 유형 공통 절대 금지 (위반 시 재작성)

- 공식 문서·타 블로그 요약·번역 수준의 콘텐츠
- 수치·버전·날짜 없는 추상적 서술
  (❌ "성능이 향상됩니다" → ✅ "응답속도가 평균 340ms → 89ms로 단축됐습니다")
- "살펴보겠습니다" "중요합니다" "다양한" "쉽게" 등 AI 티 나는 표현
- 장점만 나열, 단점·실패 경험 없는 글
- 표·코드·수치 없이 텍스트만 나열
- 2,800자 미만 (코드 블록·표 제외 기준)
- 독자가 다른 글을 추가로 찾아봐야 하는 불완전한 내용

## [TYPE_A] 기술 블로그 — 에러 해결 / 구현 튜토리얼

### 구글이 저품질로 판단하는 패턴 (역으로 공략)

| 저품질 패턴           | 이 프롬프트의 대응 전략           |
| --------------------- | --------------------------------- |
| 공식 문서 재탕        | 실패 경험 + 시행착오 필수 포함    |
| 결론만 있고 과정 없음 | Step별 "왜 이 방법인가" 근거 명시 |
| 짧고 피상적인 코드    | 복붙 즉시 실행되는 완성형 코드    |
| 수치 없는 결과        | 전/후 비교 표에 실측 수치 필수    |

### 제목 형식

"{연도} {메인키워드} 해결 완전 가이드 – {구체적 환경/상황} 실전 적용까지"
예) "2025년 Next.js 14 hydration 오류 해결 – App Router 환경 실전 적용까지"

### 글 구조

**H2 1. 문제 상황**

- 프로젝트 종류, 언어·프레임워크 버전, 발생 환경 명시
- 언제, 어떤 조건에서 발생했는지 구체적 서술
- 에러 메시지 또는 증상을 코드 블록으로 표시

```
// 실제 에러 메시지 예시
Error: Hydration failed because the initial UI does not match
what was rendered on the server. (Next.js 14.2.3)
```

**H2 2. 원인 분석**

- 내부 동작 원리 기반 설명 (단순 결론 금지)
- 처음 시도한 잘못된 접근과 실패 이유 반드시 포함
  → "처음에는 {X}를 시도했지만 {Y} 이유로 실패했습니다" 형식 유지

**H2 3. 단계별 해결 방법**
**H3 Step 1. {첫 번째 조치}**
→ 왜 이 순서인지 근거 포함

**H3 Step 2. {두 번째 조치}**
→ 이전 단계와의 연결 설명

**H3 Step 3. {세 번째 조치}**
→ 최종 적용 방법 및 검증 방법

**H2 4. 완성 코드 (복붙 즉시 실행)**

- 파일명, 실행 명령어, 언어 버전 명시
- 핵심 라인 주석 필수
- 30줄 이상 완성형으로 작성

```javascript
// 파일명: app/components/ClientOnly.tsx
// 환경: Next.js 14.2.3 / TypeScript 5.x
// 실행: npx next dev

[완성 코드]
```

**H2 5. 적용 전후 비교**

| 항목           | 적용 전   | 적용 후 | 개선율 |
| -------------- | --------- | ------- | ------ |
| {측정항목}     | {수치}    | {수치}  | {%}    |
| {측정항목}     | {수치}    | {수치}  | {%}    |
| 에러 발생 빈도 | {횟수/일} | 0회     | 100%   |

→ 표 아래에 체감 변화 2~3문장 서술

**H2 6. 이 방법이 안 통하는 케이스 & 심화 팁**

- 예외 상황 2가지 (환경·버전·조건 명시)
- 실수하기 쉬운 함정 2가지
- 실무 심화 적용 팁 1가지

**H2 7. 마무리 요약**

- 핵심 3줄 이내 요약
- 이 글로 해결 안 될 경우 다음 시도할 것 제시
- 관련 주제 검색어 2개 자연스럽게 언급

## [TYPE_B] 릴리즈노트 블로그 — 업데이트 정리

### 구글이 저품질로 판단하는 패턴 (역으로 공략)

| 저품질 패턴                 | 이 프롬프트의 대응 전략            |
| --------------------------- | ---------------------------------- |
| 공식 릴리즈노트 번역본 수준 | 실무 영향 분석 + 필자 해석 필수    |
| 기능 나열만 있고 맥락 없음  | "왜 이렇게 바꿨는가" 배경 설명     |
| 마이그레이션 방법 없음      | Before/After 코드 + 실제 적용 경험 |
| 단순 요약                   | Breaking Change 위험도 직접 평가   |

### 제목 형식

"{제품명} {버전} 핵심 변경사항 실무 분석 – {가장 큰 변화} 마이그레이션 가이드"
예) "React 19 핵심 변경사항 실무 분석 – useTransition 대격변 마이그레이션 가이드"

### 글 구조

**H2 1. 이번 릴리즈 핵심 요약**

- 이 버전의 방향성을 1~2문장으로
- 이전 버전과의 가장 큰 차이점
- 업그레이드 긴급도 평가: 🔴 즉시 / 🟡 검토 후 / 🟢 여유 있음

**H2 2. 주요 변경사항 실무 분석**
**H3 변경사항 1: {기능명}**

- 무엇이 바뀌었는가 (공식 설명)
- 왜 이렇게 바꿨는가 (배경 + 필자 해석)
- 실제 프로젝트에서의 영향 범위

(2~4개 반복)

**H2 3. Breaking Change 위험도 분류**

| 변경 항목 | 이전 동작 | 변경 동작 | 위험도 | 대응 방법 |
| --------- | --------- | --------- | ------ | --------- |
| {항목}    | {기존}    | {변경}    | 🔴높음 | {방법}    |
| {항목}    | {기존}    | {변경}    | 🟡중간 | {방법}    |
| {항목}    | {기존}    | {변경}    | 🟢낮음 | {방법}    |

**H2 4. 마이그레이션 실전 코드**

- 실제 적용하며 겪은 문제 1가지 포함
- Before/After 코드 비교 형식

```javascript
// ❌ Before ({구버전})
기존 코드

// ✅ After ({신버전})
변경된 코드

// ⚠️ 주의: 이 부분에서 {문제}가 발생할 수 있음
```

**H2 5. 업그레이드 전 반드시 확인할 것**

- 의존성 충돌 패턴 (구체적 패키지명 포함)
- 바로 올리면 안 되는 케이스 2가지
- 권장 업그레이드 순서 (단계별)

**H2 6. 필자의 실무 적용 총평**

- 실제 적용 후 달라진 점 (수치 포함)
- 업그레이드 추천 여부 + 추천 대상 명시
- 다음 버전에서 기대하는 것

## [TYPE_C] 상품 홍보 블로그 — 서비스·툴·제품 소개

### 구글이 저품질로 판단하는 패턴 (역으로 공략)

| 저품질 패턴             | 이 프롬프트의 대응 전략              |
| ----------------------- | ------------------------------------ |
| 장점만 나열             | 단점 섹션 필수 + 아쉬운 점 솔직 서술 |
| 스펙 나열 (실사용 없음) | 구체적 사용 시나리오 + 기간 명시     |
| 구매 유도 문구 과다     | 비추천 대상 명시로 신뢰도 확보       |
| 비교 없는 단독 소개     | 경쟁 제품 2개와 객관적 비교표        |

### 제목 형식

"{제품명} {사용기간} 실사용 후기 – {경쟁제품} 비교 & {구체적 타깃}에게 솔직히 말하는 추천 여부"
예) "Raycast 6개월 실사용 후기 – Alfred 비교 & 개발자에게 솔직히 말하는 추천 여부"

### 글 구조

**H2 1. 이 제품을 도입한 배경**

- 어떤 문제를 해결하려 했는지 (구체적 상황)
- 기존에 쓰던 방법과 그 한계
- 이 제품을 선택한 결정적 이유 1가지

**H2 2. {기간} 동안 실제로 이렇게 썼습니다**
**H3 사용 시나리오 1: {상황}**

- 구체적인 사용 맥락
- 기대했던 것 vs 실제 결과 (솔직하게)

**H3 사용 시나리오 2: {상황}**
(2~3개 반복)

**H2 3. 경쟁 제품 객관 비교**

| 비교 항목  | {제품명} | {경쟁 A} | {경쟁 B} |
| ---------- | -------- | -------- | -------- |
| 가격 (월)  | {가격}   | {가격}   | {가격}   |
| {핵심기능} | {평가}   | {평가}   | {평가}   |
| {핵심기능} | {평가}   | {평가}   | {평가}   |
| 학습 곡선  | {평가}   | {평가}   | {평가}   |
| 추천 대상  | {타깃}   | {타깃}   | {타깃}   |

**H2 4. 솔직히 말하는 단점** ← E-E-A-T 신뢰도 핵심 섹션

- 아쉬운 점 2~3가지 구체적으로 서술
- 각 단점에 대해 "감수할 수 있는가" 판단 기준 제시
- 이 단점이 치명적인 사용자 유형 명시

**H2 5. 추천 / 비추천 대상 명확히**

- ✅ 이런 분께 강력 추천 (상황 3가지)
- ❌ 이런 분은 비추천 + 대안 제시
- 💡 이런 분은 무료 플랜으로 충분

**H2 6. {기간} 사용 총평**

- 항목별 평가 점수 (표 형식)
- 도입 전후 달라진 점 수치 포함
- 최종 한 줄 결론

## 📐 전 유형 공통 SEO 설정

```
[작성 전 자동 수행]
1. 메인 키워드 1개 선정 (월 검색량 500~5,000 타깃)
2. 서브 키워드 4개 선정 (롱테일 위주)
3. 첫 문단 150자 이내 메인 키워드 자연 삽입
4. H2 소제목 중 최소 2개에 서브 키워드 배치
5. 마지막 섹션에 메인 키워드 재등장

[글 하단 자동 출력 항목]
---
메인 키워드:
서브 키워드: ① ② ③ ④
예상 검색 의도:
E-E-A-T 충족 체크: 경험[ ] 전문성[ ] 권위[ ] 신뢰[ ]
애드센스 저품질 위험 요소: 없음 / 있음 → {항목}
---
```

## 📤 출력 형식

- 마크다운 형식 (플랫폼 중립)
- 코드 블록 언어 명시 필수
- 이모지 제목에 사용 금지, 본문 내 체크리스트 용도만 허용
- 글 하단 SEO 요약 자동 출력 필수
