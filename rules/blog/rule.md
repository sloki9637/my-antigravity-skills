## 작성 목표

구글 애드센스 "가치 없는 콘텐츠" 정책 통과를 기준으로, E-E-A-T(경험·전문성·권위·신뢰)가 드러나는 원본 글을 작성한다.
동일 주제를 검색한 독자가 이 글 하나만으로 이해부터 실행까지 끝낼 수 있어야 한다.

- 글 유형: {TYPE_A | TYPE_B | TYPE_C}
- 글 주제: {직접 입력}

## 작성 전 필수 입력값

작성 시작 전에 아래 항목을 반드시 채운다.

1. 대상 독자: {예: 3년차 프론트엔드 개발자}
2. 실행 환경: {OS, 언어/프레임워크 버전, 배포 환경}
3. 문제/목표 한 줄: {무엇을 해결/검증하는지}
4. 검증 수단: {로그, 벤치마크, 측정 도구}
5. 기준 시점: {YYYY-MM-DD}

## 전 유형 공통 절대 금지 (위반 시 재작성)

- 공식 문서·타 블로그를 재정리한 2차 콘텐츠
- 타사/경쟁 제품 직접 비교(가격·기능·순위 포함)
- 버전·날짜·수치 없는 추상 설명
  - 금지: "성능이 향상됩니다"
  - 허용: "P95 응답시간이 340ms에서 89ms로 73.8% 단축"
- AI 상투 표현 반복
  - 예: "살펴보겠습니다", "중요합니다", "다양한", "쉽게"
- 장점만 나열하고 단점·실패 경험·트레이드오프 생략
- 코드·표·수치 없이 텍스트만 나열
- 본문 2,800자 미만(코드 블록·표 제외)
- 독자가 다른 자료를 추가 검색해야 실행 가능한 불완전 글

## 전 유형 공통 품질 게이트 (출고 전)

아래 항목 전부 충족 시에만 게시한다.

- [ ] 첫 문단(150자 이내)에 메인 키워드 1회 자연 삽입
- [ ] 본문에 버전·날짜·환경 정보 최소 3회 명시
- [ ] 실패 사례 1개 이상 + 실패 원인 설명 포함
- [ ] 정량 비교 표 1개 이상 포함
- [ ] 재현 가능한 코드/설정/명령어 포함
- [ ] "이 방법이 안 통하는 케이스" 섹션 포함
- [ ] 마지막 섹션에 메인 키워드 재등장

## TYPE_A 기술 블로그 (에러 해결 / 구현 튜토리얼)

### 제목 형식

"{연도} {메인 키워드} 해결 완전 가이드 - {구체적 환경/상황} 실전 적용까지"

예) "2026 Next.js 15 hydration 오류 해결 완전 가이드 - App Router 환경 실전 적용까지"

### 글 구조

**H2 1. 문제 상황**

- 프로젝트 종류, 언어·프레임워크 버전, 실행 환경 명시
- 언제, 어떤 조건에서 재현되는지 작성
- 실제 에러 메시지/로그를 코드 블록으로 제시

```text
Error: Hydration failed because the initial UI does not match
what was rendered on the server. (Next.js 15.0.2)
```

**H2 2. 원인 분석**

- 내부 동작 원리 기반으로 설명(결론만 제시 금지)
- 실패한 첫 시도와 실패 이유 필수 포함
- 문장 패턴 고정: "처음에는 {X}를 시도했지만, {Y} 때문에 실패했다"

**H2 3. 단계별 해결 방법**

**H3 Step 1. {첫 조치}**

- 왜 1단계로 수행하는지 근거

**H3 Step 2. {둘째 조치}**

- Step 1과의 연결/의존 관계

**H3 Step 3. {셋째 조치}**

- 최종 반영 방식 + 검증 절차(로그/테스트/측정)

**H2 4. 완성 코드 (복붙 즉시 실행)**

- 파일명, 런타임/언어 버전, 실행 명령어 명시
- 핵심 라인 주석 포함
- 30줄 이상 완성형 코드

```ts
// 파일명: app/components/ClientOnly.tsx
// 환경: Next.js 15.x / TypeScript 5.x
// 실행: pnpm dev

// [완성 코드]
```

**H2 5. 적용 전후 비교**

| 항목           | 적용 전   | 적용 후 | 개선율 |
| -------------- | --------- | ------- | ------ |
| {측정 항목}    | {수치}    | {수치}  | {%}    |
| {측정 항목}    | {수치}    | {수치}  | {%}    |
| 에러 발생 빈도 | {횟수/일} | 0회     | 100%   |

- 표 아래 체감 변화 2~3문장 작성

**H2 6. 이 방법이 안 통하는 케이스 & 심화 팁**

- 예외 상황 2가지(환경·버전·조건 명시)
- 실수하기 쉬운 함정 2가지
- 실무 심화 팁 1가지

**H2 7. 마무리 요약**

- 핵심 3줄 요약
- 미해결 시 다음 시도 순서 제시
- 관련 검색어 2개 자연 삽입

## TYPE_B 릴리즈노트 블로그 (업데이트 정리)

### 제목 형식

"{제품명} {버전} 핵심 변경사항 실무 분석 - {가장 큰 변화} 마이그레이션 가이드"

예) "React 20 핵심 변경사항 실무 분석 - Server Actions 변화 마이그레이션 가이드"

### 글 구조

**H2 1. 이번 릴리즈 핵심 요약**

- 버전 방향성을 1~2문장으로 정리
- 직전 버전 대비 가장 큰 차이 1개 명시
- 업그레이드 긴급도: 🔴 즉시 / 🟡 검토 후 / 🟢 여유

**H2 2. 주요 변경사항 실무 분석**

**H3 변경사항 1: {기능명}**

- 무엇이 바뀌었는가(공식 변경점)
- 왜 바뀌었는가(배경 + 필자 해석)
- 실제 영향 범위(코드/인프라/테스트)

(2~4개 반복)

**H2 3. Breaking Change 위험도 분류**

| 변경 항목 | 이전 동작 | 변경 동작 | 위험도  | 대응 방법 |
| --------- | --------- | --------- | ------- | --------- |
| {항목}    | {기존}    | {변경}    | 🔴 높음 | {방법}    |
| {항목}    | {기존}    | {변경}    | 🟡 중간 | {방법}    |
| {항목}    | {기존}    | {변경}    | 🟢 낮음 | {방법}    |

**H2 4. 마이그레이션 실전 코드**

- 실제 적용 중 발생한 문제 1개 이상 포함
- Before/After 코드 제시

```ts
// Before ({구버전})
// [기존 코드]

// After ({신버전})
// [변경 코드]

// 주의: {문제 상황}에서 {증상}이 발생할 수 있음
```

**H2 5. 업그레이드 전 반드시 확인할 것**

- 의존성 충돌 패턴(구체 패키지명 포함)
- 즉시 업그레이드 금지 케이스 2가지
- 권장 업그레이드 순서(단계형)

**H2 6. 필자의 실무 적용 총평**

- 적용 후 변화 수치 제시
- 추천 여부 + 추천 대상 명확화
- 다음 버전에서 기대하는 점 1~2개

## TYPE_C 상품 홍보 블로그 (서비스/툴/제품 소개)

### 제목 형식

"{제품명} {사용기간} 실사용 후기 - {구체적 타깃}에게 솔직히 말하는 추천 여부"

예) "Curora 3개월 실사용 후기 - 정보 탐색이 많은 개발자에게 솔직히 말하는 추천 여부"

### 글 구조

**H2 1. 도입 배경**

- 해결하려던 문제를 상황 중심으로 설명
- 기존 대안과 한계 명시
- 선택의 결정적 이유 1가지

**H2 2. {기간} 동안 실제 사용 방식**

**H3 사용 시나리오 1: {상황}**

- 사용 맥락(언제/어디서/무엇을)
- 기대 대비 실제 결과(장점 + 단점)

**H3 사용 시나리오 2: {상황}**
(2~3개 반복)

**H2 3. 도입 전후 객관 비교(자사 기준)**

| 비교 항목          | 도입 전(기존 방식) | 도입 후({제품명}) | 변화율 |
| ------------------ | ------------------ | ----------------- | ------ |
| 가격(월)           | {수치}             | {수치}            | {%}    |
| {핵심 기능 1}      | {수치/평가}        | {수치/평가}       | {%}    |
| {핵심 기능 2}      | {수치/평가}        | {수치/평가}       | {%}    |
| 작업 시간/응답속도 | {수치}             | {수치}            | {%}    |

**H2 4. 솔직한 단점**

- 단점 2~3가지를 사례 기반으로 작성
- 각 단점의 감수 가능 기준 제시
- 단점이 치명적인 사용자 유형 명시

**H2 5. 추천/비추천 대상 명확화**

- 추천 대상 3가지
- 비추천 대상(자사 제품 기준에서 안 맞는 사용자) 제시
- 무료 플랜으로 충분한 대상 제시

**H2 6. {기간} 사용 총평**

- 항목별 점수 표
- 도입 전후 변화 수치 포함
- 최종 결론 1문장

## 전 유형 공통 SEO 설정

### 작성 전 자동 수행

1. 메인 키워드 1개 선정(월 검색량 500~5,000)
2. 서브 키워드 4개 선정(롱테일 우선)
3. 첫 문단 150자 이내 메인 키워드 삽입
4. H2 소제목 최소 2개에 서브 키워드 배치
5. 마지막 섹션에 메인 키워드 재삽입

### 글 하단 자동 출력 템플릿

```md
---
메인 키워드:
서브 키워드: ① ② ③ ④
예상 검색 의도:
E-E-A-T 충족 체크: 경험[ ] 전문성[ ] 권위[ ] 신뢰[ ]
애드센스 저품질 위험 요소: 없음 / 있음 -> {항목}
---
```

## 출력 형식

- 마크다운 형식(플랫폼 중립)
- 코드 블록 언어 명시 필수
- 이모지 제목 사용 금지(본문 체크리스트는 허용)
- 글 하단 SEO 요약 자동 출력 필수
